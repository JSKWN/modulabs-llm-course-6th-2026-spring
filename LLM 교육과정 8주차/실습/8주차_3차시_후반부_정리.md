## 1. 스냅샷(Snapshot)과 상태 확인 — `get_state()`

### 개념
`get_state(config)`는 특정 thread의 **현재 상태를 스냅샷**으로 찍어 반환합니다.
그래프가 어디까지 실행됐는지, 메시지는 뭐가 쌓여있는지, 다음 노드는 어디인지를 한눈에 확인할 수 있습니다.

### 실습 코드

```python
# alice 스레드의 현재 상태를 스냅샷으로 가져오기
cfg_a = {'configurable': {'thread_id': 'alice'}}
snap = mem_app.get_state(cfg_a)
```

```python
snap  # StateSnapshot 객체 전체 출력 — 대화 내역이 담겨있음
```

```python
len(snap.values['messages'])  # 쌓인 메시지 수 확인
```

```python
snap.next  # 다음에 실행될 노드
# () ← 빈 튜플이면 그래프가 종료된 상태
# ('agent',) ← 이런 식이면 아직 진행 중
```

### 핵심 정리

| 속성 | 의미 |
|------|------|
| `snap.values['messages']` | 현재까지 쌓인 전체 메시지 리스트 |
| `snap.next` | 다음에 실행될 노드 이름 (빈 튜플이면 종료) |
| `snap.config` | 이 스냅샷의 설정 정보 (thread_id 등) |

## 2. 상태 이력 조회 — `get_state_history()`

### 개념
`get_state_history(config)`는 그래프가 실행되면서 거쳐간 **모든 중간 상태**를 시간 역순으로 반환합니다.
각 노드를 지날 때마다 상태가 기록되므로, "어떤 순서로 실행됐는지" 되돌아볼 수 있습니다.

### 실습 코드

```python
history = list(mem_app.get_state_history(cfg_a))
```

```python
history  # 전체 이력 출력
```

```python
len(history)  # 몇 단계를 거쳤는지
```

```python
# 가장 처음 상태 (history는 역순이므로 [-1]이 최초)
history[-1].values.get('messages', [])
```

### 활용 포인트
- 디버깅: "왜 이런 답이 나왔지?" → 중간 단계를 하나씩 확인
- 특정 시점으로 되돌리기: history에서 원하는 시점의 config를 꺼내 다시 실행 가능

## 3. 외부에서 상태 직접 주입 — `update_state()`

### 개념
`update_state(config, values)`는 **그래프를 실행하지 않고** 외부에서 state를 직접 넣어주는 방법입니다.
테스트, 디버깅, 또는 특정 상태에서 이어서 실행하고 싶을 때 유용합니다.

### 실습 코드

```python
# 새로운 thread를 만들고, 그래프 실행 없이 메시지를 직접 주입
cfg_x = {'configurable': {'thread_id': 'manual'}}
mem_app.update_state(cfg_x, {'messages': [HumanMessage(content="안녕!")]})

snap = mem_app.get_state(cfg_x)
```

```python
snap  # 주입한 메시지가 들어가 있음
```

```python
# 메시지 타입 확인
for m in snap.values['messages']:
    print(type(m).__name__)
# HumanMessage ← 우리가 직접 넣은 것
```

```python
# invoke(None)으로 주입된 상태에서 이어서 실행
r = mem_app.invoke(None, config=cfg_x)
r['messages']
```

```python
r['messages'][-1].content  # LLM이 "안녕!"에 대해 응답한 내용
```

### 핵심 정리
- `invoke(None)`: 새로운 입력 없이, 현재 state에서 그래프를 이어서 실행
- 수동으로 상태를 세팅한 뒤 그래프를 돌릴 수 있으므로, 특정 시나리오 재현에 유용
- `update_state` → `get_state`로 확인 → `invoke(None)`으로 이어서 실행하는 3단계 패턴

## 4. HITL (Human In The Loop) — `interrupt_before`

### 개념
실무에서 에이전트가 외부 API 호출, 결제, 이메일 전송 등 **되돌리기 어려운 행동**을 하기 전에
사람의 승인을 받도록 하는 패턴입니다.

`interrupt_before=['tools']`로 컴파일하면, **tools 노드 실행 직전에 그래프가 멈춥니다.**

### 실습 코드

```python
# interrupt_before로 컴파일 — tools 노드 전에 멈춤
hitl_app = b2.compile(
    checkpointer=MemorySaver(),
    interrupt_before=['tools']  # ← 이 노드 실행 전에 중단
)
```

```python
# 질문을 보내면 LLM이 도구 호출을 결정하지만, 실제 실행 전에 멈춤
cfg_h = {'configurable': {'thread_id': 'hitl-1'}}
hitl_app.invoke(
    {'messages': [HumanMessage(content='100 더하기 200은?')]},
    cfg_h
)
```

```python
# 현재 상태 확인
state = hitl_app.get_state(cfg_h)  # 스냅샷 찍기
```

```python
state  # 전체 상태 확인
```

```python
state.next
# ('tools',) ← tools 노드에서 멈춰있다는 의미
```

```python
# LLM이 호출하려던 도구와 인자 확인
pending = state.values['messages'][-1].tool_calls[0]
pending
# {'name': 'add', 'args': {'a': 100, 'b': 200}, 'id': '...'} 같은 형태
```

### 승인 후 이어서 실행

```python
# 사람이 확인하고 승인 → invoke(None)으로 중단된 지점부터 이어서 실행
r = hitl_app.invoke(None, config=cfg_h)
r['messages'][-1].content  # "300" 또는 "100 더하기 200은 300입니다" 등
```

### 흐름 요약

```
사용자: "100 더하기 200은?"
    │
    ▼
  agent 노드 실행 → LLM이 add(100, 200) 호출 결정
    │
    ▼
  ⛔ interrupt_before=['tools'] → 여기서 멈춤!
    │
    ▼
  사람이 state 확인 → pending tool_calls 검토
    │
    ▼
  승인 → invoke(None) → tools 노드 실행 → agent → 최종 답변
```

## 5. 변형: agent 노드 전에 멈추기

### 실습 코드

```python
# agent 노드 전에 멈추도록 설정
agent_break = b2.compile(
    checkpointer=MemorySaver(),
    interrupt_before=['agent']  # ← agent 실행 전에 중단
)

cfg_tmp = {'configurable': {'thread_id': 'agent-break'}}
agent_break.invoke(
    {'messages': [HumanMessage(content='아무거나 물어봄')]},
    config=cfg_tmp
)

st = agent_break.get_state(cfg_tmp)
```

```python
st.next
# ('agent',) ← agent 노드에서 멈춰있음
```

```python
st.values['messages']
# [HumanMessage(content='아무거나 물어봄')] ← agent가 아직 실행 안 됨
```

### 핵심 정리
- `interrupt_before=['tools']`: LLM은 생각했지만, 도구 실행 전에 멈춤 (가장 일반적)
- `interrupt_before=['agent']`: LLM 호출 자체를 하기 전에 멈춤
- 어떤 노드든 `interrupt_before`에 넣으면 해당 노드 실행 직전에 중단됨

## 전체 흐름 한눈에 보기

| 기능 | 메서드 | 용도 |
|------|--------|------|
| 현재 상태 확인 | `get_state(config)` | 메시지, 다음 노드 등 스냅샷 |
| 전체 이력 조회 | `get_state_history(config)` | 디버깅, 시점 되돌리기 |
| 상태 직접 주입 | `update_state(config, values)` | 테스트, 시나리오 재현 |
| 중간 멈춤 (HITL) | `compile(interrupt_before=[...])` | 사람 승인 후 진행 |
| 멈춘 지점에서 이어서 실행 | `invoke(None, config)` | 승인 후 속행 |

---

> 💡 이 네 가지(get_state, get_state_history, update_state, interrupt_before)는 모두
> `MemorySaver`(checkpointer)가 있어야 동작합니다. checkpointer 없이 컴파일하면 상태가 저장되지 않으므로
> 스냅샷, 이력, 중단 등을 사용할 수 없습니다.
