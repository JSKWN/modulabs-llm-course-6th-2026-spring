## 핵심 개념 상세 정리

### 7-1. Tool/Function Calling 기초 — @tool · bind_tools

LLM은 실시간 데이터(날씨·환율 등)에 직접 접근 불가 → 개발자가 만든 함수를 "도구"로 등록하면, LLM이 필요 시 자체 판단하여 호출.

#### 기본 흐름

```
사용자 질문 → LLM (bind_tools) → tool_calls 반환 (content 비어있음)
                                     ↓
                              개발자 함수 실행 (get_weather 등)
                                     ↓
                              결과를 ToolMessage로 LLM에 재전달
                                     ↓
                              LLM이 최종 답변 생성
```

#### @tool 데코레이터

```python
from langchain_core.tools import tool

@tool
def get_weather(city):
    """날씨를 조회하는 함수."""
    weather_db = {
        "seoul": "맑음, 22도, 습도 45%",
        "tokyo": "흐림, 19도, 습도 70%",
    }
    return weather_db.get(city, f"{city}: 정보 없음")

# 자동 생성 속성
get_weather.name         # 'get_weather'
get_weather.description  # '날씨를 조회하는 함수.'
get_weather.args         # {'city': {'title': 'City'}}
```

> `@tool` 데코레이터를 붙이면 함수명·docstring·인자를 자동 추출 → LLM이 언제 호출할지 판단하는 데 사용

#### bind_tools와 tool_calls

```python
llm_with_tools = llm.bind_tools([get_weather])

# 도구가 필요한 질문
result = llm_with_tools.invoke("서울 날씨 어때")
# result.content → '' (비어있음)
# result.tool_calls → [{'name': 'get_weather', 'args': {'city': '서울'}, 'id': '...'}]

# 도구가 필요 없는 질문
result2 = llm_with_tools.invoke("파이썬이 뭐야?")
# result2.content → '파이썬은...'
# result2.tool_calls → []  (빈 리스트)
```

#### tool_calls 실행

```python
if result.tool_calls:
    tc = result.tool_calls[0]
    tool_result = get_weather.invoke(tc['args'])
    print(tool_result)  # '맑음, 22도, 습도 45%'
```

> `tc['name']`은 문자열이므로 `.invoke()` 불가 — `tool_map` 딕셔너리로 매핑 필요

### 7-2. StructuredTool과 Pydantic 스키마

`@tool`보다 세밀한 인자 검증이 필요할 때 `StructuredTool.from_function` + Pydantic `BaseModel` 조합 사용.

#### Pydantic 스키마 정의

```python
from pydantic import BaseModel, Field

class ETFSearchInput(BaseModel):
    category: str = Field(description="ETF 카테고리")
    min_return: float = Field(default=0, le=100, description="최소 수익률")
    max_expense: float = Field(default=1.0, ge=0, le=5, description="최대 운용보수")
```

#### StructuredTool 생성

```python
from langchain_core.tools import StructuredTool

search_etf = StructuredTool.from_function(
    func=search_etf_func,
    name='search_etf',
    description='조건에 맞는 상품을 검색.',
    args_schema=ETFSearchInput,
)

search_etf.name  # 'search_etf'
search_etf.args  # {'category': {'description': 'ETF 카테고리', ...}, ...}
```

#### @tool vs StructuredTool 비교

| 방식 | 스키마 | 유효성 검증 | 적합 상황 |
|---|---|---|---|
| `@tool` | docstring + 타입 힌트에서 자동 추출 | 기본 | 단순한 도구 |
| `StructuredTool` | Pydantic `BaseModel` 명시 | `Field(ge=, le=, enum=)` 등 세밀 | 복잡한 인자, 범위 제한 필요 시 |

### 7-3. 멀티 툴 바인딩과 Tool Loop

#### 여러 도구 동시 바인딩

```python
all_tools = [get_weather, get_exchange_rate, search_etf]
llm_multi = llm.bind_tools(all_tools)

resp = llm_multi.invoke("서울과 도쿄 날씨 비교하고 환율도 알려줘")
# resp.tool_calls → get_weather×2 + get_exchange_rate×1 (3개 동시 호출)
```

#### tool_map 패턴

```python
tool_map = {t.name: t for t in all_tools}

for tc in resp.tool_calls:
    result = tool_map[tc['name']].invoke(tc['args'])
    print(f"{tc['name']}({tc['args']}) → {result}")
```

#### Tool Loop — 자동 반복 호출

```python
def tool_loop(query, tools, max_turns=5):
    tool_map = {t.name: t for t in tools}
    llm_t = llm.bind_tools(tools)
    messages = [HumanMessage(content=query)]

    for turn in range(max_turns):
        response = llm_t.invoke(messages)
        messages.append(response)

        if not response.tool_calls:
            return response  # 도구 호출 없으면 최종 답변

        for tc in response.tool_calls:
            result = tool_map[tc['name']].invoke(tc['args'])
            messages.append(ToolMessage(content=str(result), tool_call_id=tc['id']))

    return "최대 턴 초과"
```

> `max_turns`로 무한 루프 방지 — 토큰 소모 방어 로직

### 7-6. ToolWithFallback — 재시도 · 폴백 메커니즘

프로덕션 환경에서 도구 실패 시 대비한 래퍼 클래스.

```python
class ToolWithFallback:
    def __init__(self, primary, fallback=None, max_retries=2):
        self.primary = primary
        self.fallback = fallback
        self.max_retries = max_retries
        self.stats = {'primary_ok': 0, 'fallback_ok': 0, 'errors': 0}

    def invoke(self, args):
        for attempt in range(self.max_retries):
            try:
                result = self.primary.invoke(args)
                self.stats['primary_ok'] += 1
                return result
            except Exception:
                continue

        if self.fallback:
            try:
                result = self.fallback.invoke(args)
                self.stats['fallback_ok'] += 1
                return result
            except Exception:
                pass

        self.stats['errors'] += 1
        return 'no result'
```

| 실행 순서 | 조건 | 동작 |
|---|---|---|
| 1차 | primary 성공 | 즉시 반환 |
| 2차 | primary 실패, retry 횟수 남음 | primary 재시도 |
| 3차 | primary 전부 실패, fallback 있음 | fallback 실행 |
| 4차 | fallback도 실패 | 'no result' 반환, errors +1 |

### 7-7. MCP (Model Context Protocol)

#### 문제와 해결

| 상황 | 커넥터 수 | 문제 |
|---|---|---|
| MCP 없이 | N개 도구 × M개 클라이언트 = N×M | 조합 폭발 |
| MCP 적용 | N + M | 표준 프로토콜로 연결 |

#### 구성 요소

```
[MCP Server]                    [MCP Client]
 FastMCP("server-name")          ClientSession + stdio_client
 @mcp.tool() 로 도구 등록         session.list_tools() → 도구 발견
 mcp.run()으로 서버 실행           session.call_tool(name, args) → 호출
```

#### MCP Server 예시 (01_basic_server.py)

```python
from mcp.server.fastmcp import FastMCP
mcp = FastMCP("etf-basic-server")

@mcp.tool()
def search_etf(category: str, min_return: float = 0.0) -> str:
    """ETF를 카테고리와 최소 수익률로 검색."""
    results = [e for e in ETF_DB
               if e["category"] == category and e["return_1y"] >= min_return]
    return json.dumps(results, ensure_ascii=False, indent=2)

if __name__ == "__main__":
    mcp.run()  # stdio transport
```

#### MCP Client 예시 (01_basic_client.py)

```python
import asyncio
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

async def main():
    server_params = StdioServerParameters(
        command="python", args=["servers/01_basic_server.py"]
    )
    async with stdio_client(server_params) as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()                           # 핸드셰이크
            tools = (await session.list_tools()).tools            # 도구 목록
            result = await session.call_tool("search_etf",       # 도구 호출
                         arguments={"category": "해외주식", "min_return": 20})

asyncio.run(main())
```

#### MCP + LangChain 통합 (02_langchain_client.py)

MCP 도구를 LangChain `StructuredTool`로 변환하여 `bind_tools` 가능.

```python
def mcp_tool_to_langchain(mcp_tool, session, loop):
    tool_name = mcp_tool.name  # 클로저로 캡처
    def invoke_tool(**kwargs):
        result = loop.run_until_complete(
            session.call_tool(tool_name, arguments=kwargs)
        )
        return result.content[0].text
    return StructuredTool.from_function(
        func=invoke_tool, name=tool_name,
        description=mcp_tool.description,
    )
```

> async MCP 호출을 `loop.run_until_complete()`로 동기 래핑 → LangChain 호환

### 7-8. REST API 연동 — 외부 API + LLM

#### HTTP 기본

| 메서드 | 용도 | 예시 |
|---|---|---|
| GET | 데이터 조회 | 사용자 정보, 게시글 목록 |
| POST | 데이터 생성 | 새 게시글 작성 |
| PUT | 데이터 수정 | 프로필 업데이트 |
| DELETE | 데이터 삭제 | 게시글 삭제 |

| 상태 코드 | 의미 |
|---|---|
| 2xx | 성공 |
| 4xx | 클라이언트 오류 (잘못된 요청) |
| 5xx | 서버 오류 |

#### API를 @tool로 래핑

```python
import requests

@tool
def fetch_user(user_id) -> str:
    """사용자 정보를 조회."""
    r = requests.get(f"https://jsonplaceholder.typicode.com/users/{user_id}")
    if not r.ok:
        return f"error: status {r.status_code}"
    data = r.json()
    return json.dumps({"name": data["name"], "email": data["email"],
                       "phone": data["phone"]}, ensure_ascii=False)

@tool
def fetch_user_post(user_id) -> str:
    """사용자의 게시글 목록을 조회."""
    r = requests.get(f"https://jsonplaceholder.typicode.com/posts?userId={user_id}")
    posts = r.json()
    return json.dumps([{"id": p["id"], "title": p["title"]} for p in posts],
                      ensure_ascii=False)
```

#### LLM + API 도구 연동

```python
llm_api = llm.bind_tools([fetch_user, fetch_user_post])
response = llm_api.invoke("사용자 1번의 정보와 게시글을 알려줘")

tool_map = {t.name: t for t in [fetch_user, fetch_user_post]}
messages = [HumanMessage(content=query)]
messages.append(response)

for tc in response.tool_calls:
    result = tool_map[tc['name']].invoke(tc['args'])
    messages.append(ToolMessage(content=result, tool_call_id=tc['id']))

final = llm_api.invoke(messages)  # API 결과 포함된 최종 답변
```

### 7-9. API 페이지네이션 — Offset · Cursor 방식

대량 데이터를 한 번에 가져오지 않고 페이지 단위로 나눠 요청.

#### Offset 기반 (page + limit)

```python
def fetch_all_posts(page_size=10, max_pages=100):
    for page in range(1, max_pages + 1):
        r = requests.get("https://jsonplaceholder.typicode.com/posts",
                         params={"_page": page, "_limit": page_size})
        items = r.json()
        if not items:
            break
        yield from items  # Generator로 메모리 효율적 반환
```

#### Cursor 기반

```python
def cursor_paginate_simulator(total_items=25, page_size=10):
    all_data = [{"id": i, "value": f"item_{i}"} for i in range(total_items)]

    def fetch_page(cursor=None, limit=page_size):
        start = cursor if cursor else 0
        end = start + limit
        items = all_data[start:end]
        next_cursor = end if end < total_items else None
        return {"items": items, "next_cursor": next_cursor}

    return fetch_page
```

| 방식 | 파라미터 | 장점 | 단점 |
|---|---|---|---|
| Offset | `_page`, `_limit` | 구현 간단, 임의 페이지 접근 | 중간 삽입/삭제 시 중복·누락 |
| Cursor | `cursor`, `limit` | 데이터 변경에 안전 | 이전 페이지 접근 어려움 |

#### 조건부 수집 (필터 + 조기 종료)

```python
def collect_completed_todos(max_items=20, page_size=10):
    collected = []
    for page in range(1, 100):
        r = requests.get(".../todos", params={"_page": page, "_limit": page_size})
        items = r.json()
        if not items:
            break
        for item in items:
            if item['completed'] and len(collected) < max_items:
                collected.append(item)
        if len(collected) >= max_items:
            break
    return collected
```

#### build_user_profile — 여러 API 결과 통합

```python
def build_user_profile(user_id):
    profile = {"user_id": user_id}
    try:
        user = requests.get(f".../users/{user_id}").json()
        profile["name"] = user["name"]
        profile["email"] = user["email"]
        profile["city"] = user.get("address", {}).get("city", "N/A")
    except Exception:
        profile["name"] = "error"

    posts = requests.get(f".../posts?userId={user_id}").json()
    profile["post_count"] = len(posts)

    todos = requests.get(f".../todos?userId={user_id}").json()
    done = [t for t in todos if t["completed"]]
    profile["todo_completion"] = f"{len(done)}/{len(todos)}"

    return profile
```

```python
def get_by_path(data, path, default=None):
    keys = path.split('.')
    for key in keys:
        if isinstance(data, dict):
            data = data.get(key, default)
        else:
            return default
    return data
```

### 7-11. Orchestrator 패턴

여러 도구를 LLM이 자율적으로 선택·호출하며 복잡한 작업을 수행하는 루프.

```python
def orchestrate(query, max_turns=5):
    MAX_TOOL_CALLS = 10
    tools = [api_get_user, api_get_user_posts, api_get_user_todos]
    tool_map = {t.name: t for t in tools}
    llm_api = llm.bind_tools(tools)
    messages = [HumanMessage(content=query)]
    total_calls = 0

    for turn in range(max_turns):
        resp = llm_api.invoke(messages)
        messages.append(resp)

        if not resp.tool_calls:
            return resp.content  # 최종 답변

        for tc in resp.tool_calls:
            if total_calls >= MAX_TOOL_CALLS:
                messages.append(ToolMessage(
                    content="호출 한도 초과", tool_call_id=tc['id']))
                continue
            tool_obj = tool_map.get(tc['name'])
            if not tool_obj:
                messages.append(ToolMessage(
                    content=f"도구 '{tc['name']}' 없음", tool_call_id=tc['id']))
                continue
            result = tool_obj.invoke(tc['args'])
            messages.append(ToolMessage(content=str(result), tool_call_id=tc['id']))
            total_calls += 1

    return "최대 턴 초과"
```

> `MAX_TOOL_CALLS`: 과도한 API 호출 방지. 도구 미발견 시 에러 메시지를 ToolMessage로 전달 → LLM이 대응 가능

### 7-12. Custom Tool — BaseTool 상속 · 캐싱 · 상태 관리

`@tool`이나 `StructuredTool`로 부족할 때 `BaseTool`을 직접 상속하여 상태 관리.

#### APICallerTool — 호출 횟수 추적 + 캐싱

```python
from langchain_core.tools import BaseTool

class APICallerTool(BaseTool):
    name: str = "api_caller"
    description: str = "API endpoint를 호출."
    call_count: int = 0
    cache: dict = {}

    def _run(self, endpoint: str) -> str:
        if endpoint in self.cache:
            return f"[cache] {self.cache[endpoint]}"
        self.call_count += 1
        r = requests.get(endpoint)
        result = r.text[:200]
        self.cache[endpoint] = result
        return result
```

#### StockTool — Pydantic 입력 스키마 + 쿼리 로그

```python
class StockInput(BaseModel):
    ticker: str
    include_history: bool = False

class StockTool(BaseTool):
    name: str = "stock_lookup"
    description: str = "주식 시세 조회."
    args_schema: type = StockInput
    query_log: list = []

    def _run(self, ticker: str, include_history: bool = False) -> str:
        self.query_log.append(ticker)
        prices = {"AAPL": 178.5, "GOOGL": 140.2, "MSFT": 380.1}
        price = prices.get(ticker, None)
        if price is None:
            return f"{ticker}: 정보 없음"
        result = f"{ticker}: ${price}"
        if include_history:
            result += " (5일: +2.3%)"
        return result
```

#### @tool vs StructuredTool vs BaseTool

| 방식 | 상태 관리 | 입력 검증 | 복잡도 | 적합 상황 |
|---|---|---|---|---|
| `@tool` | ✗ | 기본 | 낮음 | 단순 함수 래핑 |
| `StructuredTool` | ✗ | Pydantic 검증 | 중간 | 인자 제약 필요 |
| `BaseTool` | ✓ (캐시, 카운터, 로그) | Pydantic 검증 | 높음 | 상태 추적, 캐싱 필요 |

### Tool Calling 전체 흐름

```
[사용자 질문]
    │
    ▼
llm.bind_tools([tool_a, tool_b, ...])
    │
    ▼
llm_with_tools.invoke(messages)
    │
    ├─ tool_calls 없음 → 일반 응답 반환 (content에 텍스트)
    │
    └─ tool_calls 있음 → content 비어있음
        │
        ▼
    tool_map = {t.name: t for t in tools}

        result = tool_map[tc['name']].invoke(tc['args'])
        messages.append(ToolMessage(content=result, tool_call_id=tc['id']))
        │
        ▼

```

## 고려사항 & 질문 모음

1. **@tool의 docstring이 중요한 이유**: LLM이 도구를 호출할지 판단할 때 `description`(= docstring)을 읽고 결정. docstring이 부정확하면 엉뚱한 상황에서 호출되거나, 필요할 때 호출되지 않음.

2. **tool_calls가 비었는지 확인하는 이유**: LLM은 도구가 필요 없다고 판단하면 `tool_calls=[]`을 반환하고 `content`에 직접 답변. 무조건 도구 실행 코드를 돌리면 에러 발생.

3. **tool_map 패턴**: `tool_calls[0]['name']`은 문자열이므로 `.invoke()` 불가. `{t.name: t for t in tools}` 딕셔너리로 이름 → 도구 객체 매핑 필요.

4. **ToolMessage의 tool_call_id**: LLM이 여러 도구를 동시 호출할 때, 어떤 결과가 어떤 호출에 대한 응답인지 매칭하기 위해 필요. 빠뜨리면 에러.

5. **max_turns와 MAX_TOOL_CALLS**: 도구 호출이 연쇄적으로 이어질 수 있음 (A 결과 → B 호출 → ...). 제한 없으면 토큰 소모가 폭증하므로 반드시 상한 설정.

6. **MCP의 N×M → N+M 이점**: 도구 10개, 클라이언트 5개가 있을 때, 직접 연결은 50개 커넥터 필요. MCP 표준 프로토콜을 쓰면 10+5=15개 구현으로 충분.

7. **MCP async ↔ LangChain sync 변환**: MCP는 `asyncio` 기반이지만 LangChain `StructuredTool`은 동기 함수를 기대. `loop.run_until_complete()`로 브릿지하되, 이벤트 루프 중첩 주의 (Jupyter에서는 `nest_asyncio` 필요할 수 있음).

8. **BaseTool의 캐싱 주의**: `cache: dict = {}`로 선언하면 클래스 레벨 공유가 될 수 있음. Pydantic v2에서는 `default_factory=dict` 사용 권장.

9. **Offset vs Cursor 페이지네이션**: 데이터가 실시간 변경되는 환경(SNS 피드 등)에서는 Offset 방식으로 중복/누락 발생 가능 → Cursor 방식이 안전. 정적 데이터(ETF 목록 등)는 Offset도 무방.

10. **tool_choice 사용 시 주의**: 강제 호출이므로 LLM이 무관한 입력에도 인자를 억지로 채움 → 엉뚱한 결과 가능. 특정 시나리오(항상 날씨 확인 후 답변 등)에서만 사용.
